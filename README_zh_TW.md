# 📝 Meeting Notes Skill

[English](README.md) | **繁體中文**

一個 GitHub Copilot 自訂技能，可將會議逐字稿轉換為專業、結構化的會議紀錄，並自動校正語音辨識錯誤。

## 概述

**Meeting Notes** 技能幫助您將原始會議逐字稿（特別是語音轉文字輸出）轉換為條理分明的會議紀錄。它利用可自訂的詞彙表自動校正術語、人名及縮寫中常見的語音辨識錯誤，然後使用標準化模板進行格式化。

## ✨ 功能特色

- **🔤 自動逐字稿校正** — 使用可編輯的詞彙表修正常見的語音辨識錯誤（例如："欸愛" → "AI"、"開批愛" → "KPI"）
- **👥 與會人員辨識與驗證** — 從逐字稿中提取出席者姓名，並在定稿前向使用者確認
- **📋 標準化模板格式** — 產生一致且專業的會議紀錄，包含結構化的各章節
- **🌐 多語系模板** — 支援英文及繁體中文（zh_TW）會議模板
- **📖 可編輯詞彙表** — 使用者可持續更新術語詞彙表，提升校正準確度
- **🧠 詞彙表自動學習** — 在逐字稿處理過程中自動發現新的校正規則，主動建議加入詞彙表
- **✅ 會議紀錄驗證器** — Python 腳本驗證會議紀錄的完整性、格式正確性及交叉引用一致性

## 🚀 快速開始

### 前置需求

- 支援自訂技能的 [GitHub Copilot](https://github.com/features/copilot)
- VS Code 或相容的編輯器

### 安裝步驟

1. 複製此儲存庫或將技能資料夾複製到您的 Copilot 技能目錄：
   ```bash
   git clone https://github.com/Angus-repo/meeting-notes-skill.git
   ```

2. 自訂 `references/glossary_zh_TW.md` 詞彙表，加入您團隊的：
   - 團隊成員姓名
   - 公司/產品名稱
   - 領域專用術語
   - 常見的語音辨識錯誤

3. 即可搭配 GitHub Copilot 使用！

## 📖 使用方式

### 基本逐字稿處理

向 Copilot 提供原始會議逐字稿，請求產生會議紀錄：

```
請幫我處理這個會議逐字稿：
今天開批愛檢討會議，王曉明報告了欸愛專案的進度...
```

技能將會：

1. **讀取詞彙表** 以了解校正規則
2. **校正語音辨識錯誤** 並呈現校正紀錄
3. **辨識與會人員** 並請您確認
4. **使用模板產生結構化會議紀錄**

### 更新詞彙表

您可以請 Copilot 更新詞彙表以新增術語：

```
請更新詞彙表，新增我們的產品名稱 CloudSync，常被辨識為 Cloud Sink
```

## 📄 模板

| 模板 | 語言 | 檔案 |
|------|------|------|
| English | EN | `assets/meeting-template.md` |
| 繁體中文 | zh_TW | `assets/meeting-template_zh_TW.md` |

兩個模板皆包含以下章節：
- 會議基本資訊（名稱、日期、時間、地點、主持人、記錄人）
- 與會人員名單（出席 / 請假 / 缺席）
- 會議議程
- 討論摘要含決議事項與待辦事項
- 下次會議安排
- 附件與備註

## ✅ 驗證功能

使用內建的驗證腳本檢查會議紀錄的完整性與正確性：

```bash
# 自動偵測語言進行驗證
python3 scripts/validate_notes.py path/to/meeting-notes.md

# 指定輸出語言
python3 scripts/validate_notes.py path/to/meeting-notes.md --lang zh_TW

# 輸出 JSON 格式（適用於自動化流程）
python3 scripts/validate_notes.py path/to/meeting-notes.md --json

# 包含逐字稿覆蓋率驗證
python3 scripts/validate_notes.py path/to/meeting-notes.md \
  --transcript path/to/transcript.txt \
  --glossary references/glossary_zh_TW.md
```

驗證器檢查項目：

| 檢查項目 | 說明 |
|----------|------|
| 基本資訊完整性 | 所有欄位（日期、時間、地點、主持人、記錄人）皆已填寫 |
| 日期格式 | 日期符合 YYYY-MM-DD 格式 |
| 與會人員 | 至少列出一位出席者 |
| 議程項目 | 至少有一項實際議程 |
| 討論內容 | 各議題包含討論重點與決議事項 |
| 待辦事項格式 | 每項待辦包含負責人與期限 |
| 模板結構 | 所有必要章節皆存在 |
| 交叉驗證 | 待辦負責人出現在與會人員名單中 |
| **逐字稿覆蓋率** | 逐字稿中的關鍵事實已被記錄在會議紀錄中 |

### 逐字稿覆蓋率驗證

當提供 `--transcript` 參數時，驗證器會從原始逐字稿中提取**關鍵事實**，並檢查它們是否出現在會議紀錄中：

| 事實類型 | 提取內容 |
|----------|----------|
| 👤 人名 | 詞彙表中的姓名在逐字稿中出現的部分 |
| 🔢 數字 | 百分比、金額、數量 |
| 📅 日期 | 日期、截止日、時間參照 |
| 🔨 決策 | 包含決策關鍵字的語句（決定/同意/agreed/approved） |
| 📌 行動 | 包含指派關鍵字的語句（負責/截止/assigned to） |
| 📖 術語 | 詞彙表中的技術和商業術語 |

覆蓋率門檻：
- ≥ 80% → ✅ 通過
- 50–79% → ⚠️ 警告
- < 50% → ❌ 錯誤

## 🔧 自訂設定

### 詞彙表（`references/glossary_zh_TW.md`）

詞彙表支援以下分類：

| 分類 | 範例 |
|------|------|
| 技術術語 | AI、API、CI/CD、DevOps、Kubernetes |
| 商業術語 | KPI、ROI、SOP、OKR、B2B |
| 部門名稱 | 研發部、業務部、行銷部 |
| 人名 | 中文及英文姓名 |
| 公司/產品名稱 | 自訂項目 |
| 會議相關詞彙 | 議程、決議、共識 |

> **提示：** 當有新團隊成員加入、新專案啟動或發現新的語音辨識錯誤時，請定期更新詞彙表。

## 💡 最佳實踐

1. **保持詞彙表更新** — 更好的詞彙表 = 更高的校正準確度
2. **務必驗證校正結果** — 在定稿前檢閱校正紀錄
3. **確認與會人員名單** — 不要假設自動偵測的名單是完整的
4. **一致使用模板** — 維持所有會議紀錄的專業標準
5. **新增領域專用術語** — 產業術語和內部用語能顯著提升校正效果
